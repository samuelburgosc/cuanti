# Cuanti — Design Context

## Design Context

### Users
Microemprendedores latinoamericanos (Chile, 25–70 años) que venden productos físicos — streetwear, ropa, calzado, accesorios. Usan la app desde el celular mientras están en el local, en ferias o en movimiento.

**Contexto de uso**: ruido, luz variable, pantalla pequeña, sesiones de 30–60 segundos. No están sentados frente a un escritorio. Están vendiendo.

**El usuario modelo**: alguien que vende todos los días, lleva la gestión en la cabeza + WhatsApp + cuadernos. Está cansado de esa dispersión. Quiere un lugar donde todo esté, sin complicaciones.

**La pregunta que vienen a responder**: ¿Cuánto gané hoy?
**La sensación que buscan**: alivio y claridad, no otro sistema que aprender.

---

### Brand Personality
- **3 palabras**: directo, concreto, nuestro (es de ellos, no software corporativo)
- **Metáfora del producto**: "una silla para descansar" — el usuario llega a Cuanti a dejar de complicarse, no a trabajar más
- **Tono**: amigo que sabe de números, no contador. Habla en plata, no en términos técnicos.
- **Emoción que debe evocar**: alivio y control. "Por fin entiendo mi negocio."
- **Anti-referencia explícita**: cualquier cosa que parezca ERP, SAP, Excel, Bsale, o "software contable"
- **Anti-referencia de sensación**: dashboards que abruman con información sin decir qué hacer

---

### Lenguaje — Regla Obligatoria
**Siempre español latinoamericano (tuteo). Nunca voseo argentino. Sin excepciones.**

| ✅ Correcto | ❌ Incorrecto |
|---|---|
| "ingresa tu producto" | "ingresá tu producto" |
| "¿cuánto ganaste?" | "¿cuánto ganaste?" (ok) |
| "crea tu primera venta" | "creá tu primera venta" |
| "ponle un nombre" | "ponele un nombre" |
| "puedes ver" | "podés ver" |
| "tienes" | "tenés" |

Aplica a: copy de la app, mensajes de error, empty states, onboarding, tooltips, chat con el usuario, y cualquier documento.

---

### Aesthetic Direction
- **Light mode** — paleta cálida off-white. No dark mode.
- **Mobile-first, max 430px** — diseñado para celular, no adaptado.
- **Accent amarillo `#D7F02E`** — energético, de atención. El color que dice "esto importa". Usar con moderación (regla 60-30-10).
- **Tipografía**: Archivo Black (títulos, números grandes) + Archivo (cuerpo) + DM Mono (números, SKUs, moneda)
- **Minimalismo funcional con jerarquía fuerte**: pocos elementos, mucho espacio, UNA cosa dominante por pantalla
- **Paleta de tokens**:
  - `--bg: #F5F5F2` `--sf: #FFFFFF` `--sf2: #F2F2EE` `--sf3: #EAEAE6`
  - `--bd: #E6E3DD` `--bd2: #D4D0CA`
  - `--accent: #D7F02E`
  - `--red: #D84B4B` `--blue: #2563EB` `--orange: #E46A2E` `--green: #1E9E63` `--purple: #B47FFF`
  - `--text: #171717` `--muted: #7A7A7A` `--muted2: #BCBCBC`

---

### Design Principles
1. **Una cosa domina** — cada pantalla tiene UNA cosa más importante. Todo lo demás la apoya o se esconde.
2. **La ganancia es el centro** — cada pantalla debe poder responder "¿cuánto gané?"
3. **Alivio, no tarea** — la app debe sentirse como descansar, no como trabajar más
4. **Acción en máximo 3 pasos** — flujos cotidianos (vender, registrar gasto) no pueden ser lentos
5. **Datos concretos sobre decoración** — los números son el diseño, los adornos son ruido
6. **Sin jerga técnica** — si suena a Excel o a contador, está mal
7. **El contexto manda** — la app muestra diferente según el estado del usuario (nuevo, sin ventas, con datos)

## Patrones UX — Reglas Concretas

### Empty States (estados vacíos)
Nunca mostrar "—" ni texto genérico. Cada empty state debe enseñar qué hacer.

| Estado | Qué mostrar |
|---|---|
| Sin productos Y sin ventas | Setup card amarillo: "Para ver tus ganancias, empieza acá" + botón único |
| Con productos, sin ventas | Setup card amarillo: "¡Ya tienes productos! Registra tu primera venta" + botón |
| Con ventas (estado normal) | Nada especial — mostrar datos reales |
| Búsqueda sin resultados | "No encontramos nada con ese nombre" (no "Sin resultados") |
| Pantalla Alertas vacía | "Todo en orden. Cuando haya stock bajo o novedades, aparecen acá." |
| Scanner: código no encontrado | "Producto no encontrado. ¿Es un producto nuevo?" + botón "Crear producto →" |

### Dashboard — Lógica de visibilidad por estado
```
esPrimerVez (0 productos, 0 ventas):
  ✓ Hero: "Bienvenido a Cuanti 👋" + $0 en muted + subtitle explicativo
  ✓ Setup card: único CTA visible, fondo amarillo
  ✗ Stats strip: OCULTAR (datos vacíos confunden)
  ✗ Divider: OCULTAR
  ✗ Sección "Actividad reciente": OCULTAR
  ✗ Hero clickeable: DESACTIVAR (no tiene destino útil)

soloProductos (productos, 0 ventas):
  ✓ Hero: "¿Cuánto ganaste este mes?" + $0
  ✓ Setup card: CTA "Registrar primera venta →"
  ✗ Stats strip: OCULTAR (todo en $0/—)
  ✓ Actividad: mostrar mensaje "Aún no hay ventas registradas"

normal (tiene ventas):
  ✓ Todo visible: hero clicable → análisis, stats strip, actividad reciente
```

### Alertas y acciones en Dashboard
- **Máximo 2 items** visibles en "Atención ahora". Si hay más, mostrar "y X alertas más → ver análisis"
- Jerarquía de alertas: rojo (agotado) > naranja (bajo stock) > azul (capital parado) > verde (insight)
- Nunca mostrar la sección si no hay items

### Setup Card (onboarding progresivo)
- Fondo: `var(--accent)` amarillo — no blanco, no gris
- Barra de progreso: sobre el fondo amarillo con texto oscuro
- Botón: `background: var(--text); color: var(--accent)` — inversión del esquema de color
- Desaparece automáticamente cuando el usuario tiene al menos 1 producto Y 1 venta
- Nunca mostrar junto con el botón duplicado en "Actividad reciente"

### Número de Ganancia (Hero)
- Tamaño: 66px, Archivo Black, letter-spacing -3px
- Sin card contenedora — flota directamente sobre la pantalla
- Color: `var(--text)` normal | `var(--muted2)` cuando es $0 sin datos | `var(--red)` si es negativo
- Label superior: texto variable según estado (ver tabla arriba)

### Formularios y flujos
- Máximo 3 pasos para cualquier acción cotidiana
- Labels en español simple: nunca "Nº correlativo", "lote", "CSV", "ajuste inv."
- Botón principal: siempre fondo `var(--accent)` o `var(--text)`, nunca gris como primario
- Confirmaciones destructivas: inline (segundo toque), nunca `window.confirm()`

### Navegación
- El FAB amarillo central siempre es "Vender" — la acción más frecuente
- "Análisis" y "Ventas" son tabs distintos: Análisis = tendencias/métricas, Ventas = historial transaccional
- Pantallas secundarias (Compras, Escáner, Config, etc.) nunca en el bottom nav

## Decisiones de Diseño — Aprendizajes

### Lo que funcionó
| Decisión | Por qué funciona |
|---|---|
| Hero $0 con muted color para primer uso | No asusta, no confunde — es obvio que hay que hacer algo |
| Setup card en amarillo (no en card blanca) | Contrasta, llama la atención, dice "esto es lo primero" |
| Ocultar stats strip sin datos | $0, —, — no aportan información, suman ruido |
| Alerta como strip contextual (no card) | Ocupa menos espacio, aparece solo cuando hay algo real |
| "Crear producto →" en scanner not-found | Transforma un error en una oportunidad de acción |

### Lo que se rechazó y por qué
| Intento | Por qué se rechazó | Alternativa correcta |
|---|---|---|
| Mockups en browser antes de implementar | El usuario quiere ver cambios reales en la app, no propuestas | Implementar directamente en index.html |
| Nav con tab "Más" / hamburger menu | Patrón genérico, no resuelve el problema de fondo | Mejorar empty states y onboarding contextual |
| 3 opciones A/B/C genéricas (onboarding + labels / nav simplificada / centro de comando) | Demasiado abstracto, no tiene convicción de diseño | Una sola dirección implementada con precisión |
| Dashboard como "centro de comando" con acciones sugeridas | Sobre-ingeniería — el problema era el estado vacío, no el estado con datos | Adaptive dashboard: distinto según estado del usuario |

### Patrones prohibidos en Cuanti
Además de las reglas generales de impeccable:
- ❌ `window.confirm()` para confirmaciones — usar inline con segundo toque
- ❌ Terminología técnica: SKU, lote, CSV, centro de costo, margen bruto, ajuste inv.
- ❌ Stats chips mostrando todos `—` o `$0` — ocultarlos cuando no hay datos
- ❌ Dos CTAs apuntando al mismo destino en la misma pantalla
- ❌ Cards anidadas (card dentro de card)
- ❌ Secciones con título pero sin contenido (vacías)
- ❌ Voseo argentino en cualquier texto de la app o el chat

## Known UI Problems (estado actual)

### Resueltos ✓
- ~~Dashboard con 6+ secciones de igual peso~~ → hero dominante + secciones adaptivas
- ~~"Atención ahora" mostraba hasta 5 items~~ → máximo 2
- ~~Bottom nav: "Etiquetas" confusa~~ → "Ventas" (historial)
- ~~Stats strip mostrando $0/— para usuarios nuevos~~ → oculta cuando no hay datos
- ~~Nuevo usuario ve dashboard vacío sin guía~~ → setup card en amarillo, único CTA
- ~~Terminología técnica: SKU, lote, CSV, Ajuste inv., Transferir, Nº correlativo~~ → lenguaje simple
- ~~Botón de pérdida con window.confirm()~~ → inline doble toque

### Pendientes / En progreso
- **"Análisis" vs "Ventas" en el nav** — usuarios confunden ambas tabs; "Análisis" podría llamarse "Resumen" para ser más claro
- **Pantalla Análisis** — tabs múltiples con mucha info; revisar si el usuario de 25-70 años realmente usa todas
- **Escáner como pantalla secundaria** — es una de las acciones más potentes pero está enterrada; considerar acceso más rápido desde Inicio
- **Post-venta (step 3)** — ya rediseñada pero sin datos reales de costo en muchos casos; validar con usuarios reales
- **Onboarding de texto** — el setup card guía los 2 primeros pasos, pero no hay guía in-context para flujos como "cómo agrego stock" o "qué es un canal de venta"
